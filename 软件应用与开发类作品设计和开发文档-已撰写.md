# 需求分析

本作品是一个面向校园场景的心理支持平台“心屿”，核心目标不是“替代专业诊疗”，而是把学生在日常中最常见的几个心理支持需求连成一条可落地的服务链：自助识别（心理分析）→知识学习（心理百科）→情绪表达（秘密树洞）→行为调节（放松小游戏）→人工支持（联系导师）。

从当前实现看，项目重点解决了三个现实问题。第一，入口分散。很多同类产品只做单点功能，用户需要在多个页面甚至多个系统跳转，实际使用成本高；本项目把登录、个人资料、分析、社交支持和预约咨询统一到一个站点。第二，反馈缺少连续性。平台通过“分析结果+百科建议+导师预约+树洞互动”形成闭环，避免用户停留在“看完就走”。第三，技术落地不稳定。项目在联调阶段经历了聊天接口 401/500、模型加载与依赖兼容、数据库 SQL 方言兼容等问题，最终形成了较稳的前后端协作方案。

主要面向用户为高校学生与年轻群体。核心功能包括：账号登录注册、个人信息维护、心理分析、流式 AI 陪伴对话、匿名树洞互动、导师排班查询与预约/取消、心理百科检索、解压小游戏。主要性能目标聚焦“可用优先”：页面响应流畅、主要 API 可稳定返回、关键失败场景可提示用户可理解的错误信息。

竞品层面，本项目不追求“医疗级诊断能力”，而是强调“校园可用、路径完整、交互友好、部署简单”。

# 详细设计

## 1. 前端详细设计

前端采用 Vue3 + TypeScript + Vue Router。整体是单页应用，路由按业务域划分。页面外层统一由 App 承载，公共组件包含顶部导航、登录注册弹窗、分析结果弹窗、全局消息、AI 小精灵聊天。

界面层有几个关键设计点：

1. 全局统一玻璃拟态风格，主要页面都采用半透明卡片、柔和渐变和圆角按钮，降低心理产品的压迫感。
2. 小精灵入口固定在右下，支持多路由显示，避免用户在不同页面丢失求助入口。
3. 心理分析页采用“授权确认→进度反馈→结果解释”三段式，减少突兀感。
4. 导师页采用“关键词检索 + 日历筛选”的组合筛选，而不是一次性把条件绑死，符合真实使用习惯。

前端状态设计采用 composable 统一管理关键状态（登录、资料、分析、消息、百科等），降低跨组件传参复杂度。树洞、导师、聊天等高交互页面按页面内局部状态管理，减少全局污染。

## 2. 后端详细设计

后端使用 FastAPI，按路由分模块：

1. auth：注册登录、密码校验与 token 签发。
2. profile：当前用户资料查询与更新。
3. tree：树洞帖子、评论、抱抱/同感。
4. mentor：导师排班查询、预约、我的预约、取消预约。
5. analysis：读取用户 pkl 后进行 MMIM 推理。
6. chat：代理 SiliconFlow 流式聊天。

关键实现说明：

1. 鉴权策略采用 JWT。大多数敏感接口需要 token；聊天接口按联调需求支持匿名访问，避免前端无 token 时 401 影响体验。
2. 聊天流式实现没有继续依赖 SDK，而是改为 requests 直连模型服务并转发数据块，规避依赖版本冲突导致的 500 问题。
3. 分析链路最终版本不再实时从数据库拼特征，而是直接读取 backend/pkl 下以 user_id 命名的 pkl，减少线上特征构建复杂度。
4. 后端提供数据库连接异常与 SQLAlchemy 异常的统一处理，便于快速定位部署问题。

## 4. 关键算法与技术点

本项目心理分析的核心不是调用通用黑盒接口，而是使用自研心理分析模型 MMIMTemporalModel 完成多模态风险判断。模型与流程要点如下。

### 4.1 自研心理分析模型

1. 模型名称：MMIMTemporalModel（基于 MMIM 思路，结合时序建模）。
2. 输入特征：以“周”为单位的三模态特征序列。
	1. text_feat：文本语义向量（768 维）。
	2. image_feat：图像语义向量（768 维）。
	3. behavior_feat：行为统计特征（4 维）。
3. 输出结果：二分类风险概率（Non-Clinical / Clinical）及对应标签。
4. 运行方式：后端加载本地权重 best_full_model.pth，按用户粒度进行推理。

### 4.2 端到端分析流程（支持用户主页驱动）

系统支持“用户提供媒体主页地址后进入分析闭环”的端到端流程，流程为：

1. 用户在个人资料页填写个人媒体主页地址（xhs_url）。
2. 数据采集模块按主页信息抓取用户内容并标准化（标题、正文、发帖时间、图片路径等）。
3. 采集数据进入特征处理脚本，生成按周聚合的多模态特征并写入 pkl。
4. 分析接口按用户 id 读取对应 pkl，调用自研 MMIMTemporalModel 推理。
5. 前端展示健康分、风险标签、分段建议，并支持导出报告。

说明：当前工程代码中已经包含用户主页地址字段、特征处理脚本、模型推理接口与前端展示链路，形成了完整端到端分析路径。

### 4.3 其他关键技术点

1. 文本/图像编码器支持配置开关：关闭时可退化为零向量，保证推理链路在离线环境可运行。
2. 风险解释采用分段反馈：按 10 分区间映射文案，降低“同分同句”的机械感。
3. 报告导出采用前端生成：通过页面截图生成 PDF，减少后端模板渲染负担。

# 安装及使用

## 1. 环境要求

1. Node.js 20+
2. Python 3.10 或 3.11（当前更稳，不建议直接切 3.13）
3. MySQL 8+

## 2. 安装步骤

1. 创建数据库并执行 database_schema.sql。
2. 后端安装依赖并启动 FastAPI。
3. 前端安装依赖并启动 Vite。
4. 按需配置后端 .env（数据库连接、JWT、模型服务 key）。

## 3. 典型使用流程

1. 用户注册并登录。
2. 在个人中心补充资料、上传头像。
3. 进入心理分析，授权后查看结果。
4. 在心理百科查看对应议题建议。
5. 在导师页按关键词和日期筛选后完成预约。
6. 在树洞进行匿名表达与互动。
7. 在任意支持页面通过小精灵进行即时 AI 对话。


# 项目总结

本项目最大的收获不是“功能数量”，而是把一套校园心理支持流程真正做成可运行、可联调、可解释的系统。开发过程中最典型的难点有三类：

1. 跨端联动难：前端页面改动往往会牵动后端接口与数据库字段。通过按业务域拆分路由和页面，联调效率明显提升。
2. 依赖兼容难：模型与聊天两条链路都遇到版本问题，最终选择“先稳定、再优化”的策略：聊天改 requests、分析改直接读 pkl。
3. 数据兼容难：SQL 在不同数据库版本表现不同。采用更通用的写法后，脚本可执行性明显改善。

从工程角度看，当前版本已经具备完整演示能力。后续可重点升级三方面：

1. 增加自动化测试（接口回归、前端关键交互冒烟）。
2. 引入更细粒度权限与审计日志。
3. 对心理分析增加时间序列可视化与长期趋势追踪。

总体而言，本作品已经完成从“页面原型”到“可运行产品”的转变，具备继续深化为课程项目或竞赛作品的基础。
